什么是 MCP（Model Context Protocol）

Model Context Protocol（MCP） 是一种开放标准协议，旨在解决 AI 模型与外部工具、数据源之间的连接问题。不同于传统的 API 集成方式，MCP 提供了一种标准化、模块化、可扩展的通信层，让 AI 助手能够动态地发现并调用各种工具，而无需为每个工具单独编写集成代码。

在 OpenClaw 生态中，MCP 扩展是连接 AI 助手与外部世界的桥梁。通过 MCP，你可以让 OpenClaw：

  • 实时访问你的文件系统、数据库和 API
  • 触发自动化工作流（如 Zapier、n8n）
  • 与研究工具（如 NotebookLM）深度集成
  • 调用任意自定义工具，实现业务逻辑
核心优势 MCP 的最大价值在于工具发现的自动化。当一个 MCP 服务器启动时，它会向 AI 助手声明自己支持的所有工具。AI 可以在运行时动态选择使用哪些工具，而无需预先硬编码。这与传统的手动 API 集成有本质区别。

MCP vs 传统 API 集成

让我们对比一下传统集成方式与 MCP 的差异：

维度 传统 API 集成 MCP 协议
连接方式 每个工具需要单独编写集成代码 统一协议，一次配置，全局可用
工具发现 手动维护工具列表 服务器启动时自动声明
扩展性 添加新工具需要修改核心代码 启动新 MCP 服务器即可
认证管理 每个工具独立管理 统一的安全上下文
调试难度 需要逐个排查 API 调用 MCP Inspector 提供统一调试界面

MCP 的核心概念

MCP 协议中有三个核心角色：

  • Host（宿主）：运行 AI 助手的应用程序（如 OpenClaw）
  • Client（客户端）：在宿主中运行的 MCP 客户端，负责与服务器通信
  • Server（服务器）：提供工具、数据或资源的外部服务

通信采用 JSON-RPC 2.0 格式，支持两种传输层：stdio（标准输入输出，适合本地进程）和 HTTP + SSE（适合远程服务器）。OpenClaw 默认使用 stdio 传输层，确保本地 MCP 服务器的低延迟和高可靠性。

MCP 架构解析

理解 MCP 的架构有助于你在实际使用中做出正确的配置决策。整体架构可以分为传输层、协议层和应用层三层。

传输层（Transport Layer）

MCP 支持两种传输机制：

  • stdio：通过标准输入输出进行进程间通信。适合本地运行的 MCP 服务器，延迟最低，配置最简单。
  • HTTP + SSE：使用 HTTP POST 发送请求，通过 Server-Sent Events 接收响应。适合需要远程部署或需要横向扩展的场景。

OpenClaw 在本地开发时推荐使用 stdio 模式，生产环境可根据安全性需求选择 HTTP + SSE 并配合 TLS 加密。

协议层（Protocol Layer）

MCP 协议定义了四类核心能力：

能力 描述 使用场景
Tools AI 可调用的函数，带参数定义和返回值 文件操作、API 调用、业务逻辑
Resources AI 可读取的数据源，如文件、数据库 知识库查询、配置读取
Prompts 预定义的提示模板 常用工作流封装
Samples AI 可参考的示例数据 Few-shot learning

应用层（Application Layer）

在 OpenClaw 中，MCP 客户端作为插件集成。通过 openclaw mcp 命令行工具，你可以：

# 列出当前已连接的 MCP 服务器
openclaw mcp list

# 添加新的 MCP 服务器
openclaw mcp add my-server --command "npx my-mcp-server"

# 移除 MCP 服务器
openclaw mcp remove my-server
配置存储 MCP 服务器配置保存在 ~/.openclaw/mcp.json 文件中。该文件采用 JSON 格式，包含了每个服务器的启动命令、环境变量和传输层配置。

安装与配置 MCP 服务器

OpenClaw 支持两种来源的 MCP 服务器：官方维护的服务器和社区开发的服务器。本节详细介绍安装流程和配置方法。

官方 MCP 服务器

OpenClaw 官方维护了一套高质量的 MCP 服务器，覆盖了常见的工具集成场景：

服务器 功能 安装命令
filesystem 文件系统读写操作 npx @openclaw/mcp-server-filesystem
github GitHub API 集成 npx @openclaw/mcp-server-github
slack Slack 消息发送 npx @openclaw/mcp-server-slack
database SQL 数据库查询 npx @openclaw/mcp-server-database
fetch HTTP 请求发起 npx @openclaw/mcp-server-fetch

安装配置示例

以安装 GitHub MCP 服务器为例：

# 方式一：使用 openclaw CLI 添加
openclaw mcp add github --command "npx" --args "@openclaw/mcp-server-github"

# 方式二：直接编辑配置文件
# ~/.openclaw/mcp.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-github"],
      "env": {
        "GITHUB_TOKEN": "your-personal-access-token"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-filesystem"],
      "allowedDirectories": ["/Users/xiangyu/projects", "/tmp"]
    }
  }
}

环境变量与敏感信息管理

切勿将敏感信息（如 API Token）直接写入 mcp.json。推荐使用环境变量：

# ~/.openclaw/mcp.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

然后在 shell 配置文件（~/.zshrc、~/.bashrc）中设置：

export GITHUB_TOKEN="ghp_xxxxxxxxxxxx"
安全提醒 环境变量会随进程继承传递给子进程。确保你的 shell 环境安全，避免在共享机器上使用明文 Token。推荐使用专门的密文管理工具（如 1Password CLI）来注入敏感信息。

验证安装

安装完成后，可通过以下命令验证 MCP 服务器是否正常运行：

# 检查 MCP 服务器状态
openclaw mcp status

# 测试工具调用
openclaw mcp tools list

如果服务器启动失败，OpenClaw 会显示详细的错误信息，常见的启动问题包括：

  • Node.js 版本不兼容（推荐 Node.js 18+）
  • 缺少必要的依赖包
  • 环境变量未正确设置
  • 端口被占用（使用 HTTP 传输时）

连接 Zapier / n8n 自动化

将 OpenClaw 与主流自动化平台集成，可以让 AI 助手触发复杂的业务工作流。本节介绍如何连接 Zapier 和 n8n，这两者分别代表了云端托管式和自托管式的自动化方案。

Zapier 集成

Zapier 通过 Webhook 提供触发能力。我们使用 fetch MCP 服务器来向 Zapier Webhook 发送请求：

# 添加 fetch MCP 服务器
openclaw mcp add fetch --command "npx" --args "@openclaw/mcp-server-fetch"

# ~/.openclaw/mcp.json 中的配置
{
  "mcpServers": {
    "fetch": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-fetch"],
      "env": {}
    }
  }
}

在 OpenClaw 对话中，你可以这样触发 Zap：

# 在 OpenClaw 中发送请求到 Zapier Webhook
使用 fetch 工具 {
  "url": "https://hooks.zapier.com/hooks/catch/xxx/yyy/",
  "method": "POST",
  "headers": {
    "Content-Type": "application/json"
  },
  "body": {
    "event": "ai_triggered",
    "data": "${user_input}"
  }
}

Zapier 高级配置：OAuth 认证

如果你的 Zap 需要 OAuth 认证，可以在 fetch 请求中加入 Authorization 头：

{
  "url": "https://api.zapier.com/v2/batteries/123/triggers",
  "method": "POST",
  "headers": {
    "Authorization": "Bearer ${ZAPIER_OAUTH_TOKEN}",
    "Content-Type": "application/json"
  },
  "body": {
    "input_data": "${query}"
  }
}

n8n 集成

n8n 提供了更灵活的自托管选项。通过 n8n 的 Webhook 节点，OpenClaw 可以触发任意复杂的工作流：

# n8n 工作流 Webhook URL
WEBHOOK_URL="https://your-n8n-server/webhook/ai-trigger"

# 在 OpenClaw 中调用 n8n
使用 fetch 工具 {
  "url": "${WEBHOOK_URL}",
  "method": "POST",
  "headers": {
    "Content-Type": "application/json",
    "X-n8n-Api-Key": "${N8N_API_KEY}"
  },
  "body": {
    "source": "openclaw",
    "action": "process",
    "payload": "${user_message}"
  }
}
推荐架构 在 n8n 中创建一个专门的 "OpenClaw 触发器" 工作流，使用 Webhook 节点作为入口，IF 节点进行路由分流，HTTP Request 节点调用外部 API，最后通过 Discord/Slack 节点返回结果。这种设计让工作流清晰可维护。

实际应用场景

以下是几个经典的自动化集成场景：

  • 邮件自动处理：OpenClaw 识别邮件内容，触发 n8n 工作流自动分类、存档或回复
  • CRM 数据同步：在 OpenClaw 中查询客户信息，自动更新到 Salesforce 或 HubSpot
  • 社交媒体发布：OpenClaw 生成内容，通过 Zapier 发布到 Twitter、LinkedIn
  • 监控系统告警：OpenClaw 分析告警内容，通过 Slack 通知相关人员

NotebookLM 集成

NotebookLM 是 Google 推出的 AI 驱动的研究助手，擅长理解和分析文档。将 OpenClaw 与 NotebookLM 集成，可以让 AI 助手在回答问题时借助你的私人知识库，实现检索增强生成（RAG）效果。

集成原理

NotebookLM 提供了 API 接口，允许外部应用查询其创建的知识库笔记本。集成架构如下：

# 整体数据流
用户问题 → OpenClaw → MCP Server → NotebookLM API
                                      ↓
                               查询笔记本内容
                                      ↓
                               返回相关片段
                                      ↓
OpenClaw 整合上下文 → 生成答案

获取 NotebookLM API 访问

NotebookLM API 目前处于有限预览阶段，需要申请访问权限：

  1. 访问 Google AI Studio
  2. 申请 NotebookLM API 预览计划
  3. 获取 API Key 并设置为环境变量

配置 NotebookLM MCP 服务器

# ~/.openclaw/mcp.json
{
  "mcpServers": {
    "notebooklm": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-notebooklm"],
      "env": {
        "NOTEBOOKLM_API_KEY": "${NOTEBOOKLM_API_KEY}",
        "NOTEBOOKLM_NOTEBOOK_ID": "${NOTEBOOKLM_NOTEBOOK_ID}"
      }
    }
  }
}

使用示例

配置完成后，在 OpenClaw 中你可以这样查询知识库：

# 查询笔记本中的相关内容
使用 notebooklm_query 工具 {
  "query": "OpenClaw 的 MCP 扩展如何配置",
  "max_results": 5
}

# 获取笔记本摘要
使用 notebooklm_summary 工具 {}
上下文窗口优化 NotebookLM 返回的片段可能很长。在调用 notebooklm_query 时，建议设置 max_results 为 3-5 个最相关的片段，避免超出 OpenClaw 的上下文窗口限制。

离线文档处理

如果你的笔记本中包含大量 PDF、论文或内部文档，NotebookLM 的多模态能力可以处理：

  • 上传 PDF 文档到笔记本
  • NotebookLM 自动提取文本并建立索引
  • OpenClaw 通过 MCP 接口检索相关内容
  • 结合上下文生成准确的答案

这种组合特别适合：法律研究、学术文献综述、技术文档问答等场景。

当官方和社区 MCP 服务器无法满足你的需求时，可以开发自己的 MCP 工具。OpenClaw 提供了完整的 MCP SDK，支持 TypeScript 和 Python 两种语言。

编写 MCP 服务器

以下是一个完整的 MCP 服务器示例，实现了一个"任务管理"工具：

// src/index.ts
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { CallToolRequestSchema, ListToolsRequestSchema } from "@modelcontextprotocol/sdk/types.js";

// 内存中的任务存储（实际应用中应连接数据库）
interface Task {
  id: string;
  title: string;
  completed: boolean;
  createdAt: string;
}

const tasks: Map<string, Task> = new Map();

// 创建 MCP 服务器实例
const server = new Server(
  {
    name: "my-task-manager",
    version: "1.0.0",
  },
  {
    capabilities: {
      tools: {},
    },
  }
);

// 声明工具列表 — AI 会看到这些工具签名
server.setRequestHandler(ListToolsRequestSchema, async () => {
  return {
    tools: [
      {
        name: "task_create",
        description: "创建一个新任务",
        inputSchema: {
          type: "object",
          properties: {
            title: {
              type: "string",
              description: "任务标题",
            },
          },
          required: ["title"],
        },
      },
      {
        name: "task_list",
        description: "列出所有任务",
        inputSchema: {
          type: "object",
          properties: {},
        },
      },
      {
        name: "task_complete",
        description: "标记任务为已完成",
        inputSchema: {
          type: "object",
          properties: {
            id: {
              type: "string",
              description: "任务 ID",
            },
          },
          required: ["id"],
        },
      },
    ],
  };
});

// 处理工具调用
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params;

  try {
    switch (name) {
      case "task_create": {
        const id = crypto.randomUUID();
        const task: Task = {
          id,
          title: args.title,
          completed: false,
          createdAt: new Date().toISOString(),
        };
        tasks.set(id, task);
        return {
          content: [
            {
              type: "text",
              text: `✅ 任务已创建：${task.title}（ID: ${id}）`,
            },
          ],
        };
      }

      case "task_list": {
        const taskList = Array.from(tasks.values());
        if (taskList.length === 0) {
          return {
            content: [{ type: "text", text: "📋 暂无任务" }],
          };
        }
        const text = taskList
          .map((t) => `${t.completed ? "✅" : "⬜"} [${t.id.slice(0, 8)}] ${t.title}`)
          .join("\n");
        return {
          content: [{ type: "text", text: `📋 任务列表：\n${text}` }],
        };
      }

      case "task_complete": {
        const task = tasks.get(args.id);
        if (!task) {
          throw new Error(`任务 ${args.id} 不存在`);
        }
        task.completed = true;
        return {
          content: [
            {
              type: "text",
              text: `✅ 任务已完成：${task.title}`,
            },
          ],
        };
      }

      default:
        throw new Error(`未知工具：${name}`);
    }
  } catch (error) {
    return {
      content: [
        {
          type: "text",
          text: `❌ 错误：${error instanceof Error ? error.message : String(error)}`,
        },
      ],
      isError: true,
    };
  }
});

// 启动服务器
async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("My MCP Server 已启动");
}

main();

if __name__ == "__main__":
    mcp.run()

安全最佳实践

MCP 扩展赋予了 AI 助手强大的工具调用能力，但这也意味着安全风险需要被认真对待。以下是保护你的 MCP 生态的关键安全实践。

最小权限原则

每个 MCP 服务器应该只拥有完成其任务所必需的最小权限：

  • 文件系统服务器：限定 allowedDirectories，避免访问敏感目录
  • 数据库服务器：使用只读数据库用户，限制 SQL 操作范围
  • API 集成服务器：使用 scoped API Token，只授权必要的 API 范围
# 错误的配置 — 权限过大
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-filesystem"],
      "allowedDirectories": ["/"]  // ❌ 可以访问整个文件系统
    }
  }
}

# 正确的配置 — 最小权限
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["@openclaw/mcp-server-filesystem"],
      "allowedDirectories": ["/Users/xiangyu/projects", "/tmp"]  // ✅ 只允许访问指定目录
    }
  }
}

敏感信息管理

永远不要将敏感信息硬编码在配置文件中。推荐使用以下方案之一：

方案 优点 缺点 推荐场景
环境变量 简单，无需额外依赖 需要手动管理 个人开发环境
1Password CLI 安全、版本控制外 需要 1Password 订阅 团队协作
HashiCorp Vault 企业级安全 运维成本高 生产环境
AWS Secrets Manager 与云服务集成良好 供应商绑定 AWS 生态

工具调用审计

在生产环境中，强烈建议开启 MCP 工具调用日志：

# ~/.openclaw/config.json
{
  "mcp": {
    "audit": {
      "enabled": true,
      "logFile": "/var/log/openclaw/mcp-audit.log",
      "logLevel": "info"
    }
  }
}

审计日志应记录：

  • 调用时间戳
  • 使用的工具名称
  • 输入参数（脱敏后）
  • 执行结果或错误信息
  • 会话标识符

网络隔离

对于使用 HTTP + SSE 传输的远程 MCP 服务器：

  • 务必使用 HTTPS 加密通信
  • 配置防火墙规则，限制可访问的 IP 范围
  • 考虑使用 VPN 隧道代替公网暴露
  • MCP 服务器之间不应有横向信任关系

输入验证

在自定义 MCP 工具中，始终验证输入参数：

// 使用 Zod 进行运行时验证
import { z } from "zod";

const CreateTaskSchema = z.object({
  title: z.string().min(1).max(200),
});

// 在处理请求时验证
const result = CreateTaskSchema.safeParse(args);
if (!result.success) {
  throw new Error(`无效输入：${result.error.message}`);
}
高危警告 切勿在 MCP 工具中直接执行未验证的用户输入作为系统命令（shell eval、exec）。这可能导致远程代码执行漏洞。如果必须执行系统命令，使用严格的白名单机制，并始终对输入进行转义和验证。

常见问题与排错

服务器启动失败

如果 MCP 服务器无法启动，按以下步骤排查：

  1. 检查命令是否正确：确认 command 和 args 配置无误
  2. 验证 Node.js 版本：node --version 确保为 18+
  3. 检查依赖安装：npm ls 查看是否有缺失依赖
  4. 查看错误日志：运行 openclaw mcp status 获取详细错误信息

工具调用超时

工具调用超过默认 30 秒超时时：

  • 检查 MCP 服务器是否陷入死循环或阻塞
  • 对于长时间运行的操作，考虑实现异步模式
  • 调整 timeout 配置（谨慎使用）

上下文窗口耗尽

如果 OpenClaw 返回上下文窗口错误：

  • 减少 MCP 工具返回的数据量
  • 使用 max_results 参数限制返回条目
  • 开启 OpenClaw 的上下文压缩功能
