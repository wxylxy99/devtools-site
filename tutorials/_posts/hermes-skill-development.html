什么是 Skill？

在 Hermes Agent 框架中，Skill（技能） 是一种程序性记忆（Procedural Memory）机制。它以 Markdown 文件的形式存在，包含了一组可复用的操作步骤、工作流程和参考信息。当 Hermes 在对话中遇到匹配的任务时，它会自动调用相应的 Skill 来指导自己完成工作。

这与传统的 Tool（工具）有本质区别。Tool 是预先编译好的 Python 代码，通过 registry.register() 注册到系统中，在每次运行时都会被加载进 Agent 的工具集中。而 Skill 则是一种更灵活的知识载体——它是给 Agent 读的"操作手册"，告诉 Agent 如何在特定场景下正确地组合使用各种工具。

Skill 的核心价值体现在以下几个方面：

  • 知识沉淀：当 Agent 学会了一种复杂的工作流程（比如如何为某个特定平台配置 CI/CD），这个流程可以被保存为一个 Skill，在未来的相似任务中复用。
  • 环境适配：不同团队、不同项目有不同的工作规范。Skill 让 Agent 能够学习和适应这些特定规范，而不是用一套通用但不够精准的方式工作。
  • 上下文感知：Skill 通过 metadata 中的 tags 和触发条件，让 Agent 能够在正确的时机调用正确的知识。
  • 协作共享：Skills Hub 允许开发者将自己编写的 Skill 分享给社区，其他人可以一键安装使用。
💡 关键理解 Skill 文件本身是纯文本（Markdown），但它可以引用同目录下的脚本文件、配置文件和参考文档。这些被引用的文件构成了 Skill 的完整工作环境。

Skill 分为两种来源：

  • 内置技能（Built-in Skills）：随 hermes-agent 源码一起发布，保存在 skills/ 目录下。这些是官方维护的高质量技能，如 gif-search、notion、claude-code 等。
  • Hub 技能（Hub Skills）：从 Hermes Skills Hub 安装的第三方技能，保存在 ~/.hermes/skills/ 目录下（如果是 profile 模式，则在对应 profile 的 skills 目录下）。

SKILL.md 文件结构详解

每个 Skill 的核心是一个名为 SKILL.md 的文件，它采用了YAML Frontmatter + Markdown 正文的双层结构。这种设计让 Skill 同时拥有了机器可读的元数据（用于搜索、过滤、触发匹配）和人类可读的文档内容（用于指导 Agent 的行为）。

整体结构概览

---
name: skill-name
description: 简短描述，说明这个技能的作用
version: 1.0.0
author: 作者名
license: MIT
prerequisites:
  env_vars: [ENV_VAR_1, ENV_VAR_2]
  commands: [command1, command2]
metadata:
  hermes:
    tags: [tag1, tag2, tag3]
    requires_toolsets: [toolset_name]
    fallback_for_toolsets: [another_toolset]
    related_skills: [other-skill-1, other-skill-2]
---

# 技能标题

简短介绍段落，说明这个技能做什么。

## When to Use

描述在什么情况下应该调用这个技能。

## Procedure

详细步骤说明...

## Verification

如何验证操作是否成功...
---
📋 结构要点 SKILL.md 的 YAML Frontmatter 必须放在文件最开头，用 --- 分隔符包裹。分隔符之间是标准 YAML 格式的键值对。正文部分使用标准 Markdown 格式，包含多个以 ## 开头的章节。

YAML Frontmatter 元数据

必需字段

字段 类型 说明
name string 技能的唯一标识名称，使用 kebab-case（短横线命名）。这是 Skills Hub 中的安装目标名。
description string 简短描述，控制在 1-2 句话内。用于 Skills Hub 列表展示和 Agent 决策判断。
version string 语义化版本号，格式为 主.次.修订（如 1.0.0）。

可选字段

字段 类型 说明
author string 技能作者，可以是用户名或组织名。
license string 开源许可证，常见值：MIT、Apache-2.0、GPL-3.0。
homepage string 相关文档或项目主页的 URL。
prerequisites.env_vars array 技能依赖的环境变量名列表（如 [NOTION_API_KEY, TENOR_API_KEY]）。
prerequisites.commands array 技能依赖的系统命令（如 [curl, jq]）。
metadata.hermes.tags array 标签列表，用于分类和搜索（如 [GIF, Media, Search, Tenor]）。
metadata.hermes.requires_toolsets array 技能需要存在的 toolset 列表。当这些 toolset 不可用时，技能不会显示。
metadata.hermes.fallback_for_toolsets array 技能的替代条件。当列出的 toolset 不可用时，显示此技能作为替代方案。
metadata.hermes.related_skills array 相关的其他技能名称列表。

metadata.hermes.requires_toolsets 和 fallback_for_toolsets 的区别

这两个字段共同构成了 Hermes Agent 的条件化技能显示机制。理解它们的区别对于编写能够在不同环境下正常工作的 Skill 至关重要。

requires_toolsets 表示正向依赖——只有当指定的 toolset 存在时，这个 Skill 才会出现在可用列表中。如果某个用户没有启用某个特定的 toolset，那么依赖它的 Skill 就不会显示，避免给用户带来困惑。

fallback_for_toolsets 表示替代关系——当指定的 toolset 不存在时，显示这个 Skill 作为替代方案。这在实现功能降级时非常有用。例如，当用户的系统没有安装 browser toolset 时，可以显示一个基于 curl 的替代 Skill。

场景示例：Tenor GIF 搜索

metadata:
  hermes:
    tags: [GIF, Media, Search, Tenor, API]
    # 这个 Skill 使用 curl 和 jq，不需要特殊的 toolset
    # 所以没有 requires_toolsets

# 假设还有一个 browser 版本的 GIF 搜索 Skill：
metadata:
  hermes:
    tags: [GIF, Media, Search, Tenor, API]
    fallback_for_toolsets: [browser]
    # 当 browser toolset 不可用时，显示这个 curl 版本

Skill 触发机制

Hermes Agent 并不只是简单地将所有 Skill 一股脑地塞给大语言模型。它有一个智能加载机制，会根据当前对话的上下文、用户输入的意图、以及系统配置来决定哪些 Skill 应该被加载。

触发方式一：用户手动加载（/skill 命令）

在交互式会话中，用户可以通过 /skill <name> 命令显式地加载一个 Skill。这个命令会将 Skill 的内容作为用户消息（user message）注入到对话上下文中，而不是系统提示词（system prompt）中。这是刻意设计的——因为用户消息不会破坏已经在进行的上下文缓存（prompt caching），而系统提示词的改变则会导致缓存失效。

# 在 Hermes 交互式会话中
/skill notion        # 加载 Notion API 技能
/skill claude-code  # 加载 Claude Code 编排技能

触发方式二：自动上下文加载

Hermes Agent 的 agent/skill_commands.py 模块会在会话初始化时扫描 ~/.hermes/skills/ 目录，建立 Skill 索引。当用户的消息与某个 Skill 的标签（tags）或描述（description）匹配时，Agent 会自动建议加载相关的 Skill。

触发方式三：CLI 命令行预加载（--skills 标志）

使用 -s 或 --skills 标志可以在启动时预加载指定的 Skill：

hermes chat -s notion,gif-search -q "在 Notion 中创建一个页面"
# -s 标志预加载 notion 和 gif-search 两个技能

触发方式四：Skills Hub 搜索安装

通过 hermes skills search 命令可以在 Skills Hub 中搜索技能，找到后使用 hermes skills install 安装。安装后的 Skill 会被添加到本地索引中，下一次相关对话时就有可能被触发。

Skill 与 System Prompt 的关系

理解 Skill 是如何被整合到 Agent 的上下文中的非常重要。Skill 的内容不是作为系统提示词的一部分被加载，而是作为附加的上下文参考注入到对话历史中。

当一个 Skill 被加载时，它的内容会被格式化为一条用户消息，这条消息会被添加到对话历史中。当 LLM 处理这条消息时，它会"看到"Skill 中的指令，并据此决定后续的行动。这种设计的优势在于：

  • 不破坏缓存：系统提示词在会话期间保持不变，已经缓存好的上下文不需要重新计算。
  • 按需加载：只有当特定 Skill 被实际使用时，它的内容才会占用上下文窗口。
  • 可叠加：用户可以在同一个会话中加载多个 Skill，它们会按加载顺序依次被添加到上下文中。

完整 Skill 示例

让我们来看几个真实的 Skill 示例，这些示例都来自 Hermes Agent 的内置技能库。

示例一：GIF Search（Tenor API）

这是一个简洁的 API 调用类 Skill，展示了如何通过 curl 和 jq 与外部 API 交互。

---
name: gif-search
description: Search and download GIFs from Tenor using curl. No dependencies beyond curl and jq. Useful for finding reaction GIFs, creating visual content, and sending GIFs in chat.
version: 1.1.0
author: Hermes Agent
license: MIT
prerequisites:
  env_vars: [TENOR_API_KEY]
  commands: [curl, jq]
metadata:
  hermes:
    tags: [GIF, Media, Search, Tenor, API]
---

# GIF Search (Tenor API)

Search and download GIFs directly via the Tenor API using curl. No extra tools needed.

## Setup

Set your Tenor API key in your environment (add to `~/.hermes/.env`):

```bash
TENOR_API_KEY=***
```

Get a free API key at https://developers.google.com/tenor/guides/quickstart — the Google Cloud Console Tenor API key is free and has generous rate limits.

## Prerequisites

- `curl` and `jq` (both standard on macOS/Linux)
- `TENOR_API_KEY` environment variable

## Search for GIFs

```bash
# Search and get GIF URLs
curl -s "https://tenor.googleapis.com/v2/search?q=thumbs+up&limit=5&key=${TENOR_API_KEY}" | jq -r '.results[].media_formats.gif.url'

# Get smaller/preview versions
curl -s "https://tenor.googleapis.com/v2/search?q=nice+work&limit=3&key=${TENOR_API_KEY}" | jq -r '.results[].media_formats.tinygif.url'
```

## Download a GIF

```bash
# Search and download the top result
URL=$(curl -s "https://tenor.googleapis.com/v2/search?q=celebration&limit=1&key=${TENOR_API_KEY}" | jq -r '.results[0].media_formats.gif.url')
curl -sL "$URL" -o celebration.gif
```

## API Parameters

| Parameter | Description |
|-----------|-------------|
| `q` | Search query (URL-encode spaces as `+`) |
| `limit` | Max results (1-50, default 20) |
| `key` | API key (from `$TENOR_API_KEY` env var) |
| `media_filter` | Filter formats: `gif`, `tinygif`, `mp4`, `tinymp4`, `webm` |
| `contentfilter` | Safety: `off`, `low`, `medium`, `high` |
| `locale` | Language: `en_US`, `es`, `fr`, etc. |

## Available Media Formats

Each result has multiple formats under `.media_formats`:

| Format | Use case |
|--------|----------|
| `gif` | Full quality GIF |
| `tinygif` | Small preview GIF |
| `mp4` | Video version (smaller file size) |
| `tinymp4` | Small preview video |
| `webm` | WebM video |
| `nanogif` | Tiny thumbnail |

## Notes

- URL-encode the query: spaces as `+`, special chars as `%XX`
- For sending in chat, `tinygif` URLs are lighter weight
- GIF URLs can be used directly in markdown: `![alt](url)`

示例二：Notion API（复杂操作类）

这个 Skill 展示了如何实现一个完整的 API 集成，包括页面创建、数据库查询、属性更新等多种操作。

---
name: notion
description: Notion API for creating and managing pages, databases, and blocks via curl. Search, create, update, and query Notion workspaces directly from the terminal.
version: 1.0.0
author: community
license: MIT
metadata:
  hermes:
    tags: [Notion, Productivity, Notes, Database, API]
    homepage: https://developers.notion.com
prerequisites:
  env_vars: [NOTION_API_KEY]
---

# Notion API

Use the Notion API via curl to create, read, update pages, databases (data sources), and blocks. No extra tools needed — just curl and a Notion API key.

## Prerequisites

1. Create an integration at https://notion.so/my-integrations
2. Copy the API key (starts with `ntn_` or `secret_`)
3. Store it in `~/.hermes/.env`:
   ```
   NOTION_API_KEY=***
   ```
4. **Important:** Share target pages/databases with your integration in Notion (click "..." → "Connect to" → your integration name)

## API Basics

All requests use this pattern:

```bash
curl -s -X GET "https://api.notion.com/v1/..." \
  -H "Authorization: Bearer *** \
  -H "Notion-Version: 2025-09-03" \
  -H "Content-Type: application/json"
```

## Common Operations

### Search

```bash
curl -s -X POST "https://api.notion.com/v1/search" \
  -H "Authorization: Bearer *** \
  -H "Notion-Version: 2025-09-03" \
  -H "Content-Type: application/json" \
  -d '{"query": "page title"}'
```

### Create Page in a Database

```bash
curl -s -X POST "https://api.notion.com/v1/pages" \
  -H "Authorization: Bearer *** \
  -H "Notion-Version: 2025-09-03" \
  -H "Content-Type: application/json" \
  -d '{
    "parent": {"database_id": "xxx"},
    "properties": {
      "Name": {"title": [{"text": {"content": "New Item"}}]},
      "Status": {"select": {"name": "Todo"}}
    }
  }'
```

### Query a Database

```bash
curl -s -X POST "https://api.notion.com/v1/data_sources/{data_source_id}/query" \
  -H "Authorization: Bearer *** \
  -H "Notion-Version: 2025-09-03" \
  -H "Content-Type: application/json" \
  -d '{
    "filter": {"property": "Status", "select": {"equals": "Active"}},
    "sorts": [{"property": "Date", "direction": "descending"}]
  }'
```

## Property Types

Common property formats for database items:

- **Title:** `{"title": [{"text": {"content": "..."}}]}`
- **Rich text:** `{"rich_text": [{"text": {"content": "..."}}]}`
- **Select:** `{"select": {"name": "Option"}}`
- **Multi-select:** `{"multi_select": [{"name": "A"}, {"name": "B"}]}`
- **Date:** `{"date": {"start": "2026-01-15", "end": "2026-01-16"}}`
- **Checkbox:** `{"checkbox": true}`
- **Number:** `{"number": 42}`
- **URL:** `{"url": "https://..."}`
- **Email:** `{"email": "user@example.com"}`
- **Relation:** `{"relation": [{"id": "page_id"}]}`

示例三：Claude Code 编排（Agent 协作类）

这个 Skill 展示了如何编排外部 AI Agent，包含了完整的 tmux 集成方案和多种使用模式。

---
name: claude-code
description: Delegate coding tasks to Claude Code (Anthropic's CLI agent). Use for building features, refactoring, PR reviews, and iterative coding. Requires the claude CLI installed.
version: 2.2.0
author: Hermes Agent + Teknium
license: MIT
metadata:
  hermes:
    tags: [Coding-Agent, Claude, Anthropic, Code-Review, Refactoring, PTY, Automation]
    related_skills: [codex, hermes-agent, opencode]
---

# Claude Code — Hermes Orchestration Guide

Delegate coding tasks to Claude Code (Anthropic's autonomous coding agent CLI) via the Hermes terminal.

## Two Orchestration Modes

### Mode 1: Print Mode (`-p`) — Non-Interactive (PREFERRED for most tasks)

Print mode runs a one-shot task, returns the result, and exits. No PTY needed.

```bash
terminal(command="claude -p 'Add error handling to all API calls in src/' --allowedTools 'Read,Edit' --max-turns 10", workdir="/path/to/project", timeout=120)
```

### Mode 2: Interactive PTY via tmux — Multi-Turn Sessions

Interactive mode gives you a full conversational REPL. Requires tmux orchestration.

```bash
# Start a tmux session
terminal(command="tmux new-session -d -s claude-work -x 140 -y 40")

# Launch Claude Code inside it
terminal(command="tmux send-keys -t claude-work 'cd /path/to/project && claude' Enter")

# Wait for startup, then send your task
terminal(command="sleep 5 && tmux send-keys -t claude-work 'Refactor the auth module' Enter")

# Monitor progress by capturing the pane
terminal(command="sleep 15 && tmux capture-pane -t claude-work -p -S -50")
```

## CLI Subcommands

| Subcommand | Purpose |
|------------|---------|
| `claude -p "query"` | Print mode (non-interactive, exits when done) |
| `claude -c` | Continue the most recent conversation |
| `claude -r "id"` | Resume a specific session |
| `claude auth status` | Check login status |
💡 Skill 写作要点 观察这三个示例的共同模式：每个 Skill 都包含 Prerequisites（前置条件）、Setup（配置步骤）、核心操作和参数参考。这种结构让 Agent 能够快速定位所需信息，而不必阅读冗长的说明。

引用文件与脚本

一个完整的 Skill 并不只有 SKILL.md 文件。它通常还包含多个引用文件（References）和辅助脚本，这些文件与 SKILL.md 放在同一个目录下。

目录结构规范

skills/
└── productivity/
    └── notion/
        ├── SKILL.md              # 核心技能定义
        └── references/
            ├── block-types.md    # Notion Block 类型参考
            └── database-schema.md # 数据库 schema 参考

引用文件的路径通过 metadata.hermes.references 字段（或在正文中直接引用）被关联到 Skill 上。这些文件提供了更详细的技术参考，让 Agent 在执行复杂操作时能够查阅。

references 目录示例

以 Notion Skill 为例，它的 references/ 目录下有 block-types.md 文件，详细列出了所有支持的 Notion Block 类型及其 JSON 结构：

# Notion Block Types Reference

## Paragraph
```json
{
  "object": "block",
  "type": "paragraph",
  "paragraph": {
    "rich_text": [{"text": {"content": "Hello world"}}]
  }
}
```

## Heading 1
```json
{
  "object": "block",
  "type": "heading_1",
  "heading_1": {
    "rich_text": [{"text": {"content": "Title"}}]
  }
}
```

## Bulleted List Item
```json
{
  "object": "block",
  "type": "bulleted_list_item",
  "bulleted_list_item": {
    "rich_text": [{"text": {"content": "List item"}}]
  }
}
```

## Code Block
```json
{
  "object": "block",
  "type": "code",
  "code": {
    "rich_text": [{"text": {"content": "print('hello')"}}],
    "language": "python"
  }
}
```

## Image
```json
{
  "object": "block",
  "type": "image",
  "image": {
    "type": "external",
    "external": {"url": "https://..."}
  }
}
```

辅助脚本文件

某些 Skill 可能会包含 shell 脚本或 Python 脚本作为辅助工具。这些脚本通常放在 Skill 目录的 scripts/ 子目录下：

skills/
└── custom/
    └── my-skill/
        ├── SKILL.md
        └── scripts/
            ├── setup.sh          # 环境设置脚本
            └── process_data.py   # 数据处理脚本

在 SKILL.md 的 Procedure 章节中，可以这样引用这些脚本：

## Procedure

1. Run the setup script to configure the environment:
   ```bash
   bash `SKILL_DIR`/scripts/setup.sh
   ```

2. Process the data using the helper script:
   ```bash
   python `SKILL_DIR`/scripts/process_data.py --input ./data.json
   ```
⚠️ 重要：路径引用 在 Skill 中引用脚本时，使用 `SKILL_DIR` 这个特殊的路径标记。Hermes Agent 会自动将其替换为 Skill 目录的实际路径。这确保了 Skill 在任何安装位置下都能正确找到其引用文件。

脚本的安全考虑

包含脚本的 Skill 在安装时会经过安全扫描（skills_guard）。扫描器会检查脚本中是否存在潜在的危险操作，如：

  • 可疑的网络请求模式
  • 文件系统修改操作（如 rm -rf）
  • 环境变量泄露风险
  • 路径遍历漏洞

如果扫描发现问题，安装过程会被阻断并给出详细的安全报告。

Skill 与 Tool 的本质区别

这是理解 Hermes Agent 架构的一个核心问题。很多初学者会混淆 Skill 和 Tool，但实际上它们解决的是完全不同层面的问题。

定义层面的区别

维度 Tool Skill
本质 可执行的代码函数 指导性文档（Markdown）
加载方式 在 Agent 初始化时注册到 Tool Registry 在对话中按需加载到上下文
修改方式 需要修改 Python 代码，重新导入 直接编辑 Markdown 文件即可
抽象层级 原子级操作（如读文件、执行命令） 流程级编排（如何组合使用 Tool）
错误处理 代码层面的异常捕获和返回 文档层面的步骤指导和验证方法
生命周期 进程级别，随 Agent 启动加载 会话级别，按需注入上下文

技术实现对比

Tool 的注册机制

Tool 是通过 tools/registry.py 中的 registry.register() 来注册的。这是一个典型的函数装饰器模式的变体：

# tools/my_tool.py
from tools.registry import registry

def my_tool(param: str, task_id: str = None) -> str:
    """实际的工具逻辑"""
    result = do_something(param)
    return json.dumps({"success": True, "data": result})

# 在模块加载时注册
registry.register(
    name="my_tool",
    toolset="my_toolset",
    schema={
        "name": "my_tool",
        "description": "描述这个工具做什么",
        "parameters": {
            "type": "object",
            "properties": {
                "param": {"type": "string", "description": "参数描述"}
            },
            "required": ["param"]
        }
    },
    handler=lambda args, **kw: my_tool(**args, **kw),
    check_fn=lambda: bool(os.getenv("REQUIRED_API_KEY")),
    requires_env=["REQUIRED_API_KEY"],
)

注册后，Tool 的 schema 会被添加到 model_tools.py 的工具集中，并在 Agent 启动时被加载到 LLM 的工具列表里。

Skill 的加载机制

Skill 则完全不同。它不需要任何 Python 代码注册：

# Skill 的唯一要求：创建一个目录和 SKILL.md 文件
skills/
└── my-custom-skill/
    ├── SKILL.md
    └── references/
        └── extra-info.md

Skill 的发现和加载由 agent/skill_commands.py 处理。当用户调用 /skill my-custom-skill 时，系统会：

  1. 在 ~/.hermes/skills/ 目录中查找 my-custom-skill/SKILL.md
  2. 读取文件内容
  3. 解析 YAML Frontmatter，验证元数据
  4. 将内容格式化为用户消息，注入当前对话上下文

使用场景对比

应该使用 Tool 的场景：

  • 需要高性能、低延迟的原子操作
  • 操作需要严格的类型验证和错误处理
  • 功能需要跨会话持久化（如文件读写、数据库操作）
  • 需要与外部系统进行复杂的协议交互

应该使用 Skill 的场景：

  • 工作流程的编排和文档化
  • 特定平台的 API 使用规范
  • 需要频繁更新的操作步骤
  • 团队特有的工作规范和约定
  • 跨多个 Tool 的复杂业务逻辑

协作关系

Skill 和 Tool 实际上是互补关系，而不是替代关系。Skill 描述的是"如何做"，而 Tool 提供的是"用什么做"。一个复杂的 Skill 通常会调用多个 Tool 来完成其目标。

协作示例：部署一个 Web 应用

假设我们要部署一个 Web 应用：

  • Tool 层：提供 terminal（执行命令）、file_read（读配置）、file_write（写文件）、web_search（查文档）等原子能力。
  • Skill 层：一个 deployment Skill 会告诉 Agent："部署到 AWS ECS 的流程是：1) 读取当前目录的 Dockerfile 2) 构建镜像 3) 推送到 ECR 4) 更新 ECS Task Definition 5) 运行新任务。"

hermes skills install 机制

Hermes Agent 提供了一套完整的 Skills Hub 生态系统，允许用户从中心化的技能市场安装第三方 Skill。安装过程经过了精心的安全设计，包括隔离审查（quarantine）、自动化扫描和用户确认等环节。

整体流程概览

hermes skills install 命令的执行流程分为以下几个阶段：

用户输入
    │
    ▼
解析 identifier（支持短名称自动解析）
    │
    ▼
从各个 Source 获取 Skill Bundle
    │  (official / github / clawhub / claude-marketplace / lobehub)
    ▼
Quarantine（隔离）
    │  将下载的文件放入 .hub/quarantine/ 目录
    ▼
Security Scan（安全扫描）
    │  使用 skills_guard 模块进行多维度检查
    ▼
Install Policy Check（安装策略检查）
    │  根据 trust_level 和 scan_result 决定是否允许安装
    ▼
User Confirmation（用户确认）
    │  显示 Skill 信息和免责声明，等待用户输入
    ▼
Install（安装）
    │  复制文件到 ~/.hermes/skills/ 目录
    ▼
Update Lock File（更新锁文件）
    │  记录已安装 Skill 的来源和版本
    ▼
Clear Prompt Cache（清除提示缓存）
    │  使新安装的 Skill 立即生效

Source（来源）适配器

Hermes Skills Hub 支持多种来源，通过 SkillSource 抽象类统一接口。主要的来源适配器包括：

Source 标识 Trust Level 说明
Official official/* builtin Nous Research 维护的内置可选技能
GitHub user/repo community / trusted 从任意 GitHub 仓库获取
ClawHub clawhub/* community OpenClaw 社区技能市场
Claude Marketplace claude-marketplace/* community Claude Code 技能市场
LobeHub lobehub/* community Lobe Chat 技能市场

identifier 解析机制

hermes skills install 支持多种格式的 identifier：

# 完整标识符（明确指定来源）
hermes skills install official/autonomous-ai-agents/claude-code
hermes skills install github/NousResearch/hermes-agent/skills/notion

# 短名称（自动搜索所有来源）
hermes skills install notion
hermes skills install gif-search

当使用短名称时，系统会调用 unified_search() 函数在所有来源中进行搜索。如果找到多个同名 Skill，会显示一个选择列表，要求用户使用完整标识符来指定要安装的具体版本。

Quarantine（隔离）机制

下载的 Skill 文件首先会被放入 ~/.hermes/skills/.hub/quarantine/ 目录进行隔离。这是一个临时目录，安装完成后会被清空。隔离机制的作用是：

  • 防止路径遍历攻击：在安装之前，文件只能存在于 quarantine 目录中，无法影响系统其他部分。
  • 统一扫描入口：所有要安装的文件都必须经过安全扫描，无论其来源。
  • 审计追踪：quarantine 目录中的文件会被记录到审计日志中。

安全扫描（skills_guard）

tools/skills_guard.py 模块负责对隔离区中的 Skill 进行安全扫描。扫描器会检查多个维度的风险：

# 扫描检查项（简化示意）
class ScanResult:
    verdict: str          # "clean" | "warning" | "blocked"
    findings: list        # 发现的问题列表

# 检查类型
- 路径遍历检测（检查是否存在 ../../../etc/passwd 类型的路径）
- 可疑命令检测（rm -rf, curl | sh 等危险模式）
- 网络请求模式检测（是否有隐蔽的数据外传）
- 环境变量泄露检测（是否意外暴露 API keys）
- 文件权限检测（是否创建了可执行文件）

安装策略

扫描完成后，should_allow_install() 函数会根据扫描结果和 Skill 的 trust_level 决定是否允许安装：

Trust Level 扫描结果为 Clean 扫描结果为 Warning 扫描结果为 Blocked
builtin / trusted ✅ 直接安装 ⚠️ 警告后安装 ❌ 阻止安装
community ✅ 用户确认后安装 ⚠️ 用户确认后安装 ❌ 阻止安装

Lock File（锁文件）

成功安装的 Skill 会被记录到 ~/.hermes/skills/.hub/lock.json 文件中。这个文件追踪了每个已安装 Skill 的：

{
  "skills": {
    "notion": {
      "install_path": "productivity/notion",
      "source": "official",
      "identifier": "official/productivity/notion",
      "version": "1.0.0",
      "installed_at": "2026-01-15T10:30:00Z",
      "bundle_hash": "sha256:abc123..."
    },
    "gif-search": {
      ...
    }
  }
}

Lock 文件是 Skills Hub 进行更新检查和卸载操作的依据。

GitHub 认证与 Rate Limit

从 GitHub 来源安装 Skill 时，Hermes 支持多种认证方式：

  • GitHub Token（环境变量）：设置 GITHUB_TOKEN 环境变量
  • gh CLI：如果系统安装了 gh CLI 并已完成登录，Hermes 会自动使用其认证信息
  • 未认证（匿名）：有 60 requests/hour 的速率限制
💡 提示 如果遇到 GitHub API rate limit 错误，设置 GITHUB_TOKEN 环境变量或运行 gh auth login 可以将限制提升到 5,000 requests/hour。

最佳实践

编写高质量 SKILL.md

1. 清晰的结构层次

每个 Skill 都应该遵循一致的结构：

# Skill 标题（一句话描述）

简短介绍段落（2-3 句话，说明这个技能做什么）。

## When to Use
描述在什么情况下应该使用这个技能。

## Prerequisites
列出依赖的工具、环境变量、系统命令等。

## Setup
如果需要额外配置，写在这里。

## Procedure
核心操作步骤，用编号列表。

## Verification
如何验证操作是否成功。

## Notes
注意事项、已知限制、常见问题。

## References
如果有参考文档，在这里链接。

2. 使用具体的示例

每个操作步骤都应该配有具体的代码示例。避免过于抽象的描述，因为 Agent 需要根据示例来理解具体的命令格式。

3. 包含错误处理

在 Procedure 章节中，说明可能出错的场景以及如何处理：

## 如果 API 返回 429 错误（rate limit）
等待 60 秒后重试，或者使用 `--delay` 参数降低请求频率。

## 如果认证失败
检查 `NOTION_API_KEY` 环境变量是否正确设置。

4. 善用标签（tags）

tags 是 Agent 判断何时应该加载 Skill 的重要依据。选择精确且相关的标签：

metadata:
  hermes:
    tags: [Notion, API, Productivity, Notes, Database]
    # 而不是只用 ["Notion"] 这样过于宽泛的标签

组织引用文件

按功能模块拆分

如果一个 Skill 涉及多个功能模块，将参考文档按模块拆分到不同的文件中：

skills/
└── advanced-api/
    ├── SKILL.md
    └── references/
        ├── authentication.md     # 认证相关
        ├── rate-limiting.md      # 限流相关
        ├── error-codes.md         # 错误码参考
        └── webhooks.md            # Webhook 配置

在正文中引用参考文件

在 SKILL.md 的适当位置引用参考文档：

## 认证方式

详细说明请参阅 [认证参考文档](references/authentication.md)。

## 错误处理

错误码的完整列表见 [错误码参考](references/error-codes.md)。

版本管理

为 Skill 设置语义化版本号，并遵循以下规范：

  • 主版本号（1.0.0 → 2.0.0）：当你做了不兼容的 API 变更
  • 次版本号（1.0.0 → 1.1.0）：当你添加了新功能但保持向后兼容
  • 修订号（1.0.0 → 1.0.1）：当你做了向后兼容的 bug 修复

安全注意事项

⚠️ 重要提醒 永远不要在 Skill 文件中硬编码 API 密钥、密码或其他敏感信息。使用环境变量引用，并在 SKILL.md 中说明需要设置哪些环境变量。

## Setup

```
NOTION_API_KEY=your_api_key_here
```

发布到 Skills Hub

如果你希望将自己的 Skill 分享给社区：

  1. 将 Skill 组织成标准结构（SKILL.md + 可选的 references/ 和 scripts/）
  2. 在 GitHub 上创建一个公开仓库
  3. 将 Skill 放在仓库根目录或指定目录下
  4. 使用 hermes skills tap add 测试从你的仓库安装
  5. 向 Hermes Skills Hub 提交 Pull Request 添加你的仓库索引

总结

Hermes Agent 的 Skill 系统为 AI Agent 提供了一个强大而灵活的知识管理框架。通过 SKILL.md 这种简单的 Markdown 格式，开发者可以：

  • 积累经验：将解决过的问题、学会的工作流程保存下来，供未来复用
  • 团队协作：通过 Skills Hub 分享自己的 Skill，从社区获取他人的 Skill
  • 灵活组合：Skill 与 Tool 各司其职——Tool 提供原子能力，Skill 编排复杂流程
  • 安全保障：安装任何 Skill 前都要经过安全扫描，确保不引入恶意代码

Skill 系统的设计哲学体现了 Hermes Agent 的一条核心原则：Agent 应该能够从经验中学习，并将学到的知识持久化。不同于每次都从零开始的大型系统提示词，Skill 允许细粒度、按需加载的知识注入，这既节省了上下文窗口，又提高了特定任务的表现质量。

  • 尝试编写自己的 Skill 并通过 hermes skills install 安装测试
  • 学习 Agent 开发中的 Tool 编写，与 Skill 写作结合使用
